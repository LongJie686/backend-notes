# 第3讲：高性能架构模式

## 核心结论（5条必记）

1. **单机高性能是基础** -- Reactor模型实现事件驱动，非阻塞IO处理高并发，一个线程处理多个连接
2. **集群高性能靠负载均衡** -- 通过DNS、硬件、软件多层负载均衡，将流量分发到多台服务器
3. **数据库层高性能要分离读写** -- 主库写入，从库读取，通过binlog同步，读写分离提升整体吞吐
4. **分库分表是数据量大的解决方案** -- 水平拆分解决单表数据量过大，垂直拆分解决表字段过多
5. **缓存架构设计要避免三大问题** -- 缓存穿透（查不存在）、缓存雪崩（大量过期）、缓存热点（Key过期压垮数据库）

---

## 一、单机高性能：I/O模型

### 两种I/O模式

**阻塞I/O（BIO）：**
```
每连接对应一个线程
  -> 线程阻塞等待数据
  -> 数据到达后处理
  -> 连接数受限（线程数）
```

**非阻塞I/O（NIO）：**
```
一个线程处理多个连接
  -> 轮询检查连接是否有数据
  -> 有数据则处理
  -> 连接数不受限
```

### Reactor模型

**单Reactor单线程：**
```
Reactor线程：
  1. 监听事件
  2. 分发事件
  3. 处理事件
问题：处理慢会阻塞整个Reactor
```

**单Reactor多线程：**
```
Reactor线程：
  1. 监听事件
  2. 分发事件
Worker线程池：
  3. 处理业务逻辑
优点：处理和监听分离
```

**多Reactor多线程（主从Reactor）：**
```
MainReactor：
  1. 监听连接建立
  2. 建立连接后分配给SubReactor

SubReactor：
  3. 监听已建立连接的I/O事件
  4. 分发I/O事件

Worker线程池：
  5. 处理业务逻辑

优点：Netty、Redis采用此模型
```

### Proactor模型

**与Reactor区别：**
- Reactor：线程主动检查是否有数据可读
- Proactor：内核数据准备好后通知线程（AIO）

**特点：**
- Windows支持较好（IOCP）
- Linux支持不完善
- 使用较少

## 二、集群高性能：负载均衡

### 负载均衡层次

```
客户端
  -> DNS负载均衡（地域级别）
  -> 硬件负载均衡（F5/A10）
  -> 软件负载均衡（LVS/Nginx）
  -> 应用服务器集群
```

### DNS负载均衡

**原理：**
```
同一个域名配置多个IP地址
  -> 客户端查询时返回不同IP
  -> 简单的轮询或地理位置路由
```

**优点：**
- 简单，成本低
- 就近访问（地理位置）

**缺点：**
- 不实时，DNS缓存
- 故障切换慢（TTL生效慢）
- 流量分配不精准

### 硬件负载均衡

**特点：**
- 专用设备（F5、A10）
- 性能强（百万级并发）
- 功能全（负载均衡、防火墙、SSL卸载）

**优点：**
- 性能强劲
- 稳定可靠
- 功能全面

**缺点：**
- 价格昂贵
- 扩展性差
- 学习成本高

### 软件负载均衡

**LVS（Linux Virtual Server）：**
```
工作在OSI第4层（传输层）
  -> 基于IP+端口转发
  -> 性能接近硬件负载均衡
  -> 支持NAT、DR、TUN三种模式
```

**Nginx：**
```
工作在OSI第7层（应用层）
  -> 基于HTTP内容路由
  -> 配置灵活
  -> 功能丰富（限流、缓存、SSL）
```

**对比：**

| 维度 | LVS | Nginx |
|------|-----|-------|
| 层次 | 4层 | 7层 |
| 性能 | 极高 | 高 |
| 功能 | 负载均衡 | 负载均衡+Web服务 |
| 路由 | IP+端口 | HTTP内容 |
| 扩展 | 难 | 易（模块） |

### 负载均衡算法

| 算法 | 原理 | 优点 | 缺点 | 场景 |
|------|------|------|------|------|
| 随机 | 随机选择 | 简单 | 分布不均 | 无状态服务 |
| 轮询 | 依次选择 | 分布均匀 | 不考虑性能 | 服务器性能一致 |
| 加权轮询 | 按权重分配 | 考虑性能差异 | 权重调优 | 服务器性能不同 |
| 最少连接 | 选择连接最少的 | 动态适应 | 需维护连接数 | 长连接服务 |
| 一致性哈希 | 相同请求到同一服务器 | 保持会话 | 增减服务器影响大 | 有状态服务 |

## 三、数据库层高性能

### 读写分离

**架构：**
```
应用 -> 主库（写）
     -> 从库1（读）
     -> 从库2（读）
     -> 从库3（读）
```

**同步方式：**
```
主库写入
  -> binlog记录变更
  -> 从库I/O线程读取binlog
  -> 从库SQL线程重放binlog
  -> 从库数据与主库一致
```

**延迟问题：**
```
主库写入后立即读取
  -> 可能路由到从库
  -> 从库还未同步
  -> 读到旧数据
```

**解决方案：**
- 强制读主库（写入后一段时间）
- 半同步复制（至少一个从库确认）
- 监控延迟，延迟大时读主库

### 分库分表

**垂直拆分（按业务）：**
```
单体数据库
  -> 拆分为用户库、订单库、商品库
  -> 每个库独立部署
  -> 单库压力减少
```

**水平拆分（按数据量）：**
```
单表数据量大（千万级）
  -> 拆分为多个表/库
  -> 按分片键分散数据
```

**分片策略：**

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 范围 | ID范围 | 查询范围快 | 数据分布不均 |
| Hash | Hash取模 | 数据均匀 | 范围查询慢 |
| 地理位置 | 地域分布 | 就近访问 | 需要地域属性 |

**分片键选择：**
- 查询时能定位到具体分片
- 数据分布均匀
- 避免跨片查询

## 四、缓存架构

### 缓存穿透

**问题：**
```
查询不存在的数据
  -> 缓存中没有
  -> 请求打到数据库
  -> 数据库也没有
  -> 每次都查数据库
```

**解决：**
1. **布隆过滤器**：快速判断数据是否存在
2. **缓存空值**：缓存不存在的key，设置短过期时间
3. **请求限流**：对查询不存在数据的请求限流

### 缓存雪崩

**问题：**
```
大量缓存Key同时过期
  -> 请求全部打到数据库
  -> 数据库瞬间压力暴增
  -> 数据库崩溃
```

**解决：**
1. **过期时间加随机值**：避免同时过期
2. **缓存预热**：系统启动时加载热点数据
3. **多级缓存**：本地缓存 + 分布式缓存
4. **限流降级**：缓存失效时降级

### 缓存热点

**问题：**
```
热点Key过期
  -> 大量并发请求
  -> 同时查数据库
  -> 数据库压垮
```

**解决：**
1. **互斥锁**：只有一个请求查数据库，其他等待
2. **热点数据永不过期**：异步刷新
3. **本地缓存**：热点数据本地缓存，减少分布式缓存压力

### 缓存一致性

**策略：**

| 策略 | 说明 | 一致性 | 复杂度 |
|------|------|--------|--------|
| Cache Aside | 查：先查缓存 miss查库写缓存<br>写：先更新DB再删缓存 | 最终一致 | 低 |
| Write Through | 写同时更新缓存和DB | 强一致 | 中 |
| Write Behind | 写先更新缓存，异步更新DB | 弱一致 | 高 |

**Cache Aside实现：**
```
查询：
  1. 先查缓存
  2. 缓存有则返回
  3. 缓存没有则查DB
  4. 查到的数据写入缓存
  5. 返回数据

更新：
  1. 先更新数据库
  2. 再删除缓存
```

**为什么删缓存而不是更新：**
- 并发更新时，更新缓存顺序可能错乱
- 删除缓存简单，避免脏数据
- 下次查询时重新加载

## 五、消息队列

### 为什么需要消息队列

**三大作用：**
1. **异步处理**：非核心流程异步化，提升响应速度
2. **削峰填谷**：峰值流量先入队列，平滑处理
3. **解耦**：服务间通过消息通信，降低依赖

**场景示例：**
```
用户注册：
  同步：写入用户表
  异步：发欢迎邮件、发优惠券、初始化用户数据
```

### 消息队列选型

| MQ | 优点 | 缺点 | 适用场景 |
|----|------|------|----------|
| RabbitMQ | 功能完善、可靠性高 | 吞吐量低 | 任务队列、复杂路由 |
| RocketMQ | 高吞吐、事务消息 | 运维复杂 | 电商、金融 |
| Kafka | 极高吞吐、持久化 | 实时性稍差 | 日志收集、大数据 |
| ActiveMQ | 简单、JMS规范 | 性能一般 | 传统应用 |

### 消息可靠性

**生产端：**
```
1. 发送前：消息持久化
2. 发送时：确认机制（Confirm）
3. 发送后：回调（Return）
```

**消费端：**
```
1. 手动ACK：处理成功后才确认
2. 幂等处理：重复消费不影响结果
3. 死信队列：多次失败进入死信队列
```

### 消息顺序

**为什么需要：**
- 订单：先创建后支付
- 状态机：状态变更有序

**实现：**
1. **单分区单消费者**：最简单
2. **消息ID排序**：消费者端排序
3. **分区有序**：相同业务发到同一分区

## 六、面试高频题

### 1. Reactor模型有几种变体？
单Reactor单线程、单Reactor多线程、主从Reactor -> 主从Reactor最佳：MainReactor建立连接，SubReactor处理I/O，Worker处理业务 -> Netty采用主从Reactor模型

### 2. 读写分离有什么问题？
主从延迟导致数据不一致 -> 刚写入的数据可能读不到 -> 解决：强制读主库、半同步复制、监控延迟

### 3. 分库分表后怎么处理分页查询？
无法直接用LIMIT OFFSET -> 方案1：查询分片后聚合 -> 方案2：禁止跳页，只翻下一页 -> 方案3：搜索引擎

### 4. 缓存穿透、雪崩、热点的区别？
穿透：查不存在的数据绕过缓存 -> 雪崩：大量Key同时过期 -> 热点：单个Key过期压垮数据库 -> 解决方案不同
