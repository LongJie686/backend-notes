# 第7讲：架构设计面试高频问题

## 核心结论（5条必记）

1. **面试考察深度和广度** -- 架构思维、高性能、高可用、可扩展、实战案例都会问到
2. **准备3个完整的架构案例** -- 从背景、复杂度、方案、决策、效果能完整讲述
3. **画架构图是必备能力** -- 能画出完整的架构图，标注核心组件和数据流向
4. **体现思考过程** -- 不是背诵方案，而是解释决策过程和权衡取舍
5. **量化效果很重要** -- 用数据说话：QPS提升多少、延迟降低多少

## 一、架构思维类

### 1. 什么是架构？架构和框架、设计模式有什么区别？

**架构：**
```
架构 = 系统的高层结构
  -> 模块划分
  -> 模块关系
  -> 技术选型
  -> 部署方式

关注：系统整体结构
```

**框架：**
```
框架 = 半成品
  -> 提供基础结构
  -> 填充业务代码

关注：代码结构
```

**设计模式：**
```
设计模式 = 代码层面的最佳实践
  -> 单例、工厂、观察者...

关注：类和对象的关系
```

**区别：**
| 维度 | 架构 | 框架 | 设计模式 |
|------|------|------|----------|
| 层次 | 系统 | 代码 | 类/对象 |
| 范围 | 整体结构 | 代码组织 | 解决方案 |
| 示例 | 微服务架构 | Spring MVC | 单例模式 |

### 2. 架构设计的本质是什么？

**本质：管理复杂度**

```
复杂度来源：
  -> 高性能：QPS、响应时间
  -> 高可用：故障容忍、一致性
  -> 可扩展：业务增长、团队规模

架构设计的目标：
  -> 控制复杂度
  -> 让系统可控
  -> 支持业务发展
```

### 3. 架构设计三原则是什么？

**合适原则：**
```
不要追求"最牛"的架构
  -> 只有合适的架构
  -> 3人团队不需要微服务
  -> 日活1000不需要异地多活
```

**简单原则：**
```
简单 > 复杂
  -> 能用简单方案解决的，不要搞复杂
  -> 避免过度设计
  -> 复杂度是有代价的
```

**演化原则：**
```
架构是演化出来的，不是一开始就设计出来的
  -> 先解决核心问题
  -> 随着业务发展逐步优化
  -> 不断重构和演进
```

### 4. 架构设计四步法是什么？

**第一步：识别复杂度**
```
核心复杂度是什么？
  -> 高性能？
  -> 高可用？
  -> 可扩展？

优先级排序：集中资源解决核心问题
```

**第二步：设计备选方案**
```
至少3个方案：
  -> 方案A：简单，快速
  -> 方案B：复杂，扩展性好
  -> 方案C：折中

每个方案都有道理，不要有"错误"方案
```

**第三步：评估对比**
```
评估维度：
  -> 复杂度
  -> 开发周期
  -> 扩展性
  -> 维护成本
  -> 团队匹配
```

**第四步：细化落地**
```
架构设计细化：
  -> 架构图
  -> 接口定义
  -> 数据库设计
  -> 部署方案
  -> 监控告警
```

## 二、高性能类

### 5. Reactor模型是什么？有几种变体？

**什么是Reactor：**
```
Reactor = 事件驱动模型
  -> 单线程处理多个连接
  -> I/O多路复用
  -> 非阻塞I/O
```

**三种变体：**

| 模型 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 单Reactor单线程 | 一个线程处理所有事件 | 简单 | 处理慢会阻塞 |
| 单Reactor多线程 | Reactor监听+Worker处理 | 处理和监听分离 | 连接管理复杂 |
| 主从Reactor | MainReactor建立连接<br>SubReactor处理I/O | 最佳实践 | 复杂度高 |

**主从Reactor流程：**
```
MainReactor：
  1. 监听连接建立
  2. 建立连接后分配给SubReactor

SubReactor：
  3. 监听I/O事件
  4. 分发I/O事件

Worker线程池：
  5. 处理业务逻辑
```

**应用：**
- Netty
- Redis
- Nginx

### 6. 什么时候用读写分离，什么时候用分库分表？

**读写分离：**
```
问题：读多写少，单库扛不住
  -> 主库写入
  -> 从库读取
  -> 通过binlog同步

时机：读QPS > 写QPS × 5
```

**分库分表：**
```
问题：单表数据量过大（千万级）
  -> 水平拆分：按数据量拆分
  -> 垂直拆分：按业务拆分

时机：单表数据 > 1000万
```

**决策流程：**
```
1. 优化SQL、索引（先做这个）
2. 读写分离（读多写少）
3. 缓存（热点数据）
4. 分库分表（数据量大）
```

### 7. 分库分表的分片键怎么选？

**选择原则：**
```
1. 查询时能定位到具体分片
2. 数据分布均匀
3. 避免跨片查询
```

**常用分片键：**
```
用户表：user_id
订单表：user_id（按用户查订单）
商品表：product_id
交易表：user_id + date（按用户和时间查询）
```

**反例：**
```
订单表按order_id分片
  -> 查询用户的所有订单？跨片查询
  -> 性能差
```

### 8. 缓存穿透、缓存雪崩、缓存击穿的区别和解决方案？

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 穿透 | 查询不存在的数据 | 布隆过滤器、缓存空值 |
| 雪崩 | 大量Key同时过期 | 过期时间加随机值、缓存预热 |
| 击穿 | 热点Key过期 | 互斥锁、热点数据永不过期 |

**记忆技巧：**
- 穿透：绕过缓存直击DB
- 雪崩：大量Key过期像雪崩
- 击穿：热点Key过期击穿DB

### 9. 消息队列怎么保证消息不丢失？

**三层保障：**
```
生产端：
  1. 发送前：消息持久化
  2. 发送时：Confirm机制
  3. 发送后：Callback确认

Broker端：
  1. 消息持久化
  2. 集群部署
  3. 刷盘策略

消费端：
  1. 手动ACK
  2. 业务成功才ACK
  3. 失败重试
```

### 10. 如何设计一个高性能的秒杀系统？

**核心复杂度：**
```
1. 高并发：瞬间流量暴增
2. 数据一致性：不能超卖
```

**设计方案：**
```
1. 限流：拦截超出容量的请求
2. 缓存：商品信息、库存Redis缓存
3. 异步：下单后异步处理
4. 防超卖：Redis原子扣减
5. 降级：非核心功能降级
```

**架构图：**
```
用户 -> CDN -> 限流 -> 秒杀服务 -> Redis（扣库存）
                         -> MQ（异步下单）
                         -> 订单服务
                         -> 库存服务
```

## 三、高可用类

### 11. CAP定理是什么？

**CAP含义：**
```
C (Consistency)：一致性，所有节点同时看到相同数据
A (Availability)：可用性，每个请求都能得到响应
P (Partition Tolerance)：分区容错性，网络分区时仍能运行
```

**只能满足两个：**
```
CA：理论上存在，实际不适用（网络分区必然存在）
CP：强一致，牺牲可用性（Zookeeper）
AP：高可用，牺牲一致性（Nacos、Eureka）
```

**大多数场景选AP：**
```
- 可用性优先
- 最终一致即可
- 注册中心、缓存、大多数业务系统
```

**CP场景：**
```
- 强一致性要求
- 金融、支付、库存
```

### 12. 异地多活怎么做？最大挑战是什么？

**三种异地多活：**

| 类型 | 距离 | 延迟 | 挑战 |
|------|------|------|------|
| 同城双活 | 10-50km | <5ms | 成本高 |
| 跨城异地 | 1000+km | 20-50ms | 数据同步 |
| 跨国异地 | 更远 | >100ms | 网络复杂 |

**最大挑战：数据一致性**

**解决方案：**
```
1. 数据分层：核心数据强一致，非核心最终一致
2. 单元化：按用户ID划分，一个用户固定在一个单元
3. 路由规则：用户请求路由到归属单元
4. 数据同步：异步同步非核心数据
```

### 13. 降级、熔断、限流分别是什么？适用场景？

**降级（Degradation）：**
```
定义：牺牲非核心功能，保障核心功能
场景：系统压力大，非核心功能降级
示例：推荐服务挂了，返回热门商品
```

**熔断（Circuit Breaker）：**
```
定义：下游服务异常时快速失败
场景：下游服务响应慢或错误率高
示例：库存服务熔断，下单降级
```

**限流（Rate Limiting）：**
```
定义：超出容量的请求直接拒绝
场景：流量超出系统容量
示例：秒杀场景限流
```

**使用顺序：**
```
先限流 -> 再熔断 -> 最后降级
```

## 四、可扩展类

### 14. 微服务和SOA的区别？

| 维度 | SOA | 微服务 |
|------|-----|--------|
| 粒度 | 较粗 | 较细 |
| 通信 | 重协议（SOAP） | 轻协议（REST/RPC） |
| 部署 | 统一部署 | 独立部署 |
| 数据库 | 共享 | 独立 |
| 治理 | ESB中心化 | 去中心化 |

**微服务是SOA的演进：**
```
SOA：企业级集成，ESB为中心
微服务：互联网应用，去中心化
```

### 15. 微服务拆分的原则是什么？

**核心原则：**
```
1. 高内聚低耦合
2. 按业务能力拆分
3. DDD限界上下文
4. 三个火枪手：一个服务3个人负责
```

**拆分步骤：**
```
1. 识别业务领域（用户、商品、订单...）
2. 划定服务边界
3. 定义服务接口
4. 数据库设计（独立数据库）
5. 灰度迁移
```

**避免过度拆分：**
```
3人团队拆20个服务 -> 联调成本 > 开发成本
先做好模块化，再考虑拆服务
```

## 五、实战落地类

### 16. 你做过的系统架构是怎样的？

**回答框架（STAR）：**
```
Situation（背景）：
  "我们公司是做电商的，初期单体架构..."

Task（任务）：
  "需要支撑双11大促，峰值10万QPS，同时多团队并行开发..."

Action（行动）：
  "我负责架构设计，将系统拆分为6个服务，技术栈选了..."

Result（结果）：
  "最终支撑了15万QPS，P99延迟150ms..."
```

### 17. 你们为什么选择这个方案？考虑过哪些备选方案？

**体现思考：**
```
方案A：单体 + 缓存 + 限流
  优点：简单，快速上线
  缺点：扩展性差

方案B：微服务 + MQ + 分库分表
  优点：扩展性好
  缺点：复杂度高，团队规模不够

方案C：单体 + 异步任务 + 读写分离（选择）
  优点：平衡复杂度和扩展性
  缺点：需要异步化改造

最终选择方案C，因为：
  1. 团队规模（10人）匹配
  2. 业务阶段（成长期）
  3. 时间要求（3个月上线）
```

### 18. 如果让你重新设计，你会怎么改？

**体现成长：**
```
如果重来，我会：

1. 提前引入服务治理
   问题：没有限流熔断，故障影响全局
   改进：接入Sentinel

2. 消息队列选型
   问题：用RabbitMQ性能不够
   改进：换成RocketMQ

3. 数据库分片策略
   问题：按时间分片导致热点
   改进：按user_id分片

每个改进都是实际踩坑后的总结...
```

## 六、面试技巧

### 回答问题的结构

**总-分-总结构：**
```
1. 先给出核心结论（一句话）
2. 分点展开说明（3-5点）
3. 总结或延伸（如果时间允许）
```

**示例：**
```
问：什么是微服务？

答：
微服务是一种架构风格，将应用拆分为多个小服务，
每个服务独立部署、独立数据库。

具体来说：
1. 单一职责：一个服务只做一件事
2. 独立部署：服务可独立部署和升级
3. 去中心化：数据和去中心化
4. 故障隔离：一个服务故障不影响整体

和单体架构相比，微服务能支持更大规模的团队和业务...
```

### 画图能力

**必须会画的图：**
```
1. 系统架构图：标注核心组件和关系
2. 部署架构图：服务器部署、网络拓扑
3. 数据流图：数据流转过程
4. 时序图：服务调用时序
```

### 量化效果

**用数据说话：**
```
不要说："性能提升了"
要说："QPS从1万提升到5万，P99延迟从500ms降到100ms"

不要说："解决了高可用问题"
要说："系统可用性从99.5%提升到99.95%，全年故障时间减少22小时"
```
