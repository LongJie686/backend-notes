# 第6讲：架构实战与案例分析

## 核心结论（5条必记）

1. **架构设计要用四步法** -- 识别复杂度、设计备选方案、评估对比、细化落地，避免盲目开始
2. **消息队列架构要考虑可靠性** -- 不丢、不重、有序，生产端确认、消费端幂等、分布式事务
3. **架构是演化出来的** -- Instagram从单体到微服务，淘宝从LAMP到分布式，都是逐步演化的结果
4. **支付系统设计要安全第一** -- 幂等性、事务一致性、对账机制，每一分钱都不能错
5. **写架构设计文档是基本功** -- 背景目标、架构设计、技术选型、演进路线，都要写清楚

## 一、架构设计四步法

### 第一步：识别复杂度

**为什么先识别复杂度：**
```
不知道核心矛盾是什么
  -> 盲目套用大厂方案
  -> 过度设计
  -> 事倍功半
```

**复杂度来源：**
```
高性能：QPS要求、响应时间
高可用：故障容忍、数据一致性
可扩展：业务增长、团队规模
```

**识别方法：**
```
1. 和产品/业务沟通，了解核心指标
2. 分析现有系统的瓶颈
3. 预测未来6-12个月的增长
4. 确定核心复杂度（优先级排序）
```

**示例：**
```
电商秒杀系统：
  核心复杂度1：高并发（瞬间流量暴增）
  核心复杂度2：数据一致性（不能超卖）
  次要复杂度：用户体验、数据统计
```

### 第二步：设计备选方案

**设计原则：**
- 至少3个方案
- 方案要有明显区别
- 每个方案都有道理

**方案示例：**
```
方案1：单体 + 缓存 + 限流
  -> 简单，快速上线
  -> 适合小团队

方案2：微服务 + MQ + 分库分表
  -> 复杂，扩展性好
  -> 适合大团队

方案3：单体 + 异步任务 + 读写分离
  -> 折中方案
  -> 平衡复杂度和扩展性
```

### 第三步：评估对比

**评估维度：**
| 维度 | 方案1 | 方案2 | 方案3 |
|------|-------|-------|-------|
| 复杂度 | 低 | 高 | 中 |
| 开发周期 | 2周 | 8周 | 4周 |
| 扩展性 | 差 | 好 | 中 |
| 维护成本 | 低 | 高 | 中 |
| 团队匹配 | 好 | 差 | 好 |

**评估要点：**
- 和团队规模匹配
- 和业务阶段匹配
- 和技术能力匹配

### 第四步：细化落地

**细化内容：**
```
1. 架构设计
   -> 架构图
   -> 模块划分
   -> 接口定义

2. 数据库设计
   -> 表结构
   -> 索引
   -> 分库分表策略

3. 接口设计
   -> API定义
   -> 协议选择
   -> 幂等性设计

4. 部署方案
   -> 容器化
   -> CI/CD
   -> 监控告警
```

## 二、消息队列架构设计

### 核心挑战

**三大问题：**
```
1. 消息丢失：生产端丢失、传输丢失、消费端丢失
2. 消息重复：消费失败重试、网络抖动
3. 消息积压：生产速度 > 消费速度
```

### 不丢消息

**生产端：**
```
1. 发送前：消息持久化（本地表）
2. 发送时：Confirm机制（确认Broker收到）
3. 发送后：Callback（确认消息写入队列）
4. 补偿：定时任务扫描未发送的消息
```

**Broker端：**
```
1. 消息持久化（写入磁盘）
2. 集群部署（主从复制）
3. 刷盘策略（同步刷盘/异步刷盘）
```

**消费端：**
```
1. 手动ACK（处理完成才确认）
2. 业务成功后才ACK
3. 失败重试（有限次数）
```

### 不重消息

**幂等性设计：**
```
1. 唯一ID：每个消息有唯一ID
2. 去重表：记录已处理的消息ID
3. 业务幂等：即使重复处理也不影响结果
```

**实现示例：**
```java
public void processMessage(Message message) {
    String messageId = message.getId();

    // 检查是否已处理
    if (redis.exists("processed:" + messageId)) {
        return; // 已处理，直接返回
    }

    // 处理业务
    doBusiness(message);

    // 标记已处理
    redis.set("processed:" + messageId, "1", 24h);
}
```

### 消息有序

**为什么需要有序：**
```
订单消息：
  创建 -> 支付 -> 发货
  如果顺序乱：发货了才支付
```

**实现方案：**
```
1. 单分区单消费者：最简单，但吞吐量低
2. 消息ID排序：消费者端排序
3. 分区有序：相同业务发到同一分区
```

### 消息积压处理

**原因：**
```
生产速度快，消费速度慢
  -> 队列积压
  -> 延迟增加
  -> 可能OOM
```

**解决方案：**
```
1. 增加消费者（横向扩容）
2. 优化消费逻辑（提升单消费者性能）
3. 临时扩容（应对积压）
4. 降级处理（丢弃非核心消息）
```

## 三、架构演进案例分析

### Instagram架构演进

**初期（2010）：**
```
技术栈：Django + PostgreSQL + Memcached
部署：EC2 + S3 + CloudFront
用户：100万
```

**成长期（2011）：**
```
挑战：用户快速增长（1000万+）
方案：
  -> 数据库分片（用户ID取模）
  -> Redis主从（替代Memcached）
  -> Feed系统异步化（Gearman）
```

**成熟期（2012被Facebook收购）：**
```
架构：
  -> 3000个AWS实例
  -> 数据库分片（12个分片）
  -> Redis集群（1000+节点）
  -> Instagram无限存储
```

**演进关键点：**
1. **不要过早优化**：初期简单架构
2. **数据库分片是必须**：单机无法支撑
3. **异步处理**：Feed系统异步化
4. **CDN加速**：静态资源走CDN

### 淘宝架构演进

**初期（2003）：**
```
技术栈：LAMP（Linux + Apache + MySQL + PHP）
特点：单体应用，单库
挑战：快速上线
```

**发展期（2004-2008）：**
```
挑战：业务快速增长，数据库压力
方案：
  -> 数据库垂直拆分（用户库、商品库、交易库）
  -> 数据库水平拆分（按用户ID分片）
  -> 缓存层（Memcached）
  -> 搜索独立（ISearch）
```

**成熟期（2009+）：**
```
架构：
  -> 分层架构（接入层、业务层、数据层）
  -> 服务化（HSF服务框架）
  -> 消息队列（Notify、MetaQ）
  -> 数据中间件（TDDL、DRDS）
  -> 存储中间件（OceanBase、Tair）
```

**演进关键点：**
1. **数据库拆分是基础**：先垂直后水平
2. **服务化是必经之路**：单体到服务化
3. **中间件自研**：开源无法满足需求
4. **单元化**：按用户ID路由到固定单元

## 四、支付系统架构设计

### 支付系统特点

**核心要求：**
```
1. 安全性：资金安全，防篡改
2. 一致性：账务准确，分毫不差
3. 高可用：7x24稳定
4. 可追溯：每笔交易可查询
```

### 核心流程

**支付流程：**
```
1. 用户发起支付
2. 创建支付订单
3. 调用第三方支付（微信/支付宝）
4. 支付回调
5. 更新订单状态
6. 通知业务系统
```

### 架构设计

```
接入层：API网关、鉴权、限流
  -> 业务层：支付服务、订单服务、账户服务
  -> 资金层：会计系统、清算系统
  -> 基础层：数据库、缓存、MQ
```

### 幂等性设计

**为什么需要幂等：**
```
支付回调可能重复
  -> 重复处理
  -> 重复扣款
  -> 资金损失
```

**实现方案：**
```
1. 支付订单有唯一ID
2. 回调时检查订单状态
3. 只有"待支付"状态才处理
4. 状态机保证状态流转正确
```

### 事务一致性

**挑战：**
```
支付成功后：
  -> 更新订单状态
  -> 增加账户余额
  -> 记录交易日志

任何一步失败都导致数据不一致
```

**方案：本地消息表**
```
1. 开启本地事务
2. 更新订单状态
3. 插入消息到本地消息表
4. 提交事务

定时任务：
  1. 扫描消息表
  2. 调用账户服务
  3. 成功则删除消息
  4. 失败则重试
```

### 对账机制

**为什么需要对账：**
```
技术手段保证不了100%一致
  -> 需要定期核对
  -> 发现差异
  -> 人工或自动修复
```

**对账流程：**
```
1. 从支付平台下载流水
2. 从本地数据库导出流水
3. 比对差异
4. 生成差异报告
5. 人工/自动修复
```

## 五、面试高频题

### 1. 架构设计四步法是什么？
识别复杂度 -> 设计备选方案 -> 评估对比 -> 细化落地 -> 关键是先识别核心复杂度，不要盲目开始

### 2. 消息队列怎么保证不丢消息？
生产端：消息持久化 + Confirm机制 -> Broker端：消息持久化 + 集群部署 -> 消费端：手动ACK -> 三层保障

### 3. 怎么保证接口幂等？
唯一请求ID -> 处理前检查是否已处理 -> 数据库唯一索引 -> 状态机控制状态流转 -> 幂等是重试的前提

### 4. 支付系统怎么设计？
核心要求：安全、一致、高可用 -> 幂等性设计：状态机 + 唯一ID -> 事务一致性：本地消息表 -> 对账机制：定期核对修复
