# 第5讲：可扩展架构模式

## 核心结论（5条必记）

1. **可扩展的核心思想是拆分** -- 面向流程拆分（分层）、面向服务拆分（SOA）、面向功能拆分（微服务）
2. **分层架构是最基础的模式** -- 展示层、业务层、数据层，层间解耦，每层专注自己的职责
3. **SOA通过ESB编排服务** -- 企业服务总线负责服务路由、协议转换、消息流转，但ESB可能成为瓶颈
4. **微服务是极致的拆分** -- 每个服务独立部署、独立数据库，粒度把控是关键，避免"为了微服务而微服务"
5. **微内核是插件化架构** -- 核心系统最小化，功能以插件形式扩展，保证系统稳定性和灵活性

---

## 一、可扩展的基本思想

### 为什么需要拆分

**单体系统的问题：**
```
所有代码在一个项目
  -> 代码量巨大，理解困难
  -> 编译慢，部署慢
  -> 修改一个功能影响整个系统
  -> 无法针对模块独立扩容
```

**拆分的价值：**
- **职责单一**：每个模块职责清晰
- **独立开发**：团队并行开发
- **独立部署**：快速迭代
- **弹性扩容**：针对瓶颈模块扩容
- **技术异构**：不同模块用不同技术

### 拆分维度

| 拆分方式 | 架构模式 | 核心思想 | 示例 |
|----------|---------|---------|------|
| 面向流程 | 分层架构 | 层间解耦 | MVC、DDD |
| 面向服务 | SOA | 服务编排 | ESB + 服务 |
| 面向功能 | 微服务 | 功能独立 | 用户服务、订单服务 |
| 插件化 | 微内核 | 核心+插件 | Eclipse、OSGi |

---

## 二、分层架构

### 经典三层架构

```
展示层（Presentation）
  -> 处理HTTP请求
  -> 参数验证
  -> 返回响应

业务层（Business）
  -> 业务逻辑
  -> 事务管理
  -> 调用数据层

数据层（Data）
  -> 数据访问
  -> SQL执行
  -> 缓存操作
```

### DDD分层架构

```
用户接口层（User Interface）
  -> 处理用户请求
  -> 调用应用层

应用层（Application）
  -> 用例编排
  -> 事务控制
  -> 调用领域层

领域层（Domain）
  -> 领域模型
  -> 业务规则
  -> 领域服务

基础设施层（Infrastructure）
  -> 数据访问
  -> 外部服务
  -> 技术实现
```

**优点：**
- 领域层独立，不依赖技术细节
- 业务逻辑集中在领域层
- 易于测试和维护

### 分层架构的优缺点

**优点：**
- 结构清晰，易于理解
- 职责分离，降低耦合
- 便于团队协作

**缺点：**
- 层层调用，性能损耗
- 开发效率可能降低
- 可能过度设计

**适用场景：**
- 中小型系统
- 团队规模<10人
- 业务相对简单

---

## 三、SOA架构

### SOA的核心思想

**定义：**
Service-Oriented Architecture = 面向服务的架构

**核心要素：**
```
服务（Service）
  -> 独立的业务功能
  -> 标准接口
  -> 可重用

ESB（Enterprise Service Bus）
  -> 服务总线
  -> 服务路由
  -> 协议转换
  -> 消息流转
```

### SOA架构图

```
服务消费者
  -> ESB（服务总线）
    -> 服务A（CRM）
    -> 服务B（ERP）
    -> 服务C（SCM）
```

**ESB的作用：**
1. **服务路由**：请求路由到正确的服务
2. **协议转换**：HTTP转SOAP、REST转RPC
3. **消息转换**：XML转JSON
4. **服务编排**：多个服务组合成流程

### SOA的优缺点

**优点：**
- 服务复用，避免重复建设
- 灵活编排，快速响应业务
- 技术异构，服务可用不同技术

**缺点：**
- ESB成为单点
- ESB成为瓶颈
- 服务治理复杂
- 运维成本高

**适用场景：**
- 企业级系统集成
- 遗留系统改造
- 跨部门服务共享

## 四、微服务架构

### 微服务的定义

**定义：**
```
微服务是一种架构风格
  -> 一个应用拆分为多个小服务
  -> 每个服务独立部署
  -> 服务间轻量通信
  -> 服务可选用不同技术
```

**核心特征：**
1. **单一职责**：一个服务只做一件事
2. **独立部署**：服务可独立部署和升级
3. **去中心化**：数据和去中心化
4. **故障隔离**：一个服务故障不影响整体
5. **基础设施自动化**：DevOps、CI/CD

### 微服务架构图

```
API网关（Gateway）
  -> 用户服务（独立DB）
  -> 商品服务（独立DB）
  -> 订单服务（独立DB）
  -> 支付服务（独立DB）

基础设施：
  -> 注册中心
  -> 配置中心
  -> 消息队列
  -> 监控告警
```

### 微服务拆分原则

1. **高内聚低耦合**
   - 服务内部紧密相关
   - 服务间依赖最少

2. **按业务能力拆分**
   - 用户服务：用户、权限
   - 商品服务：商品、库存
   - 订单服务：订单、支付

3. **DDD领域驱动**
   - 每个服务对应一个限界上下文
   - 领域模型清晰

4. **三个火枪手原则**
   - 一个服务由3个人负责
   - 拆分粒度与团队规模匹配

### 微服务的优缺点

**优点：**
- 独立部署：快速迭代
- 技术异构：灵活选型
- 弹性伸缩：按需扩容
- 故障隔离：影响范围小

**缺点：**
- 运维复杂：管理多个服务
- 分布式难题：一致性、调用链
- 接口兼容：版本管理
- 调试困难：跨服务问题

### 微服务 vs SOA

| 维度 | SOA | 微服务 |
|------|-----|--------|
| 粒度 | 较粗 | 较细 |
| 通信 | 重协议（SOAP） | 轻协议（REST/RPC） |
| 部署 | 统一部署 | 独立部署 |
| 数据库 | 共享 | 独立 |
| 治理 | ESB中心化 | 去中心化 |

## 五、微内核架构

### 什么是微内核

**定义：**
```
微内核架构 = 最小核心 + 插件系统

核心（Kernel）：
  -> 最小化功能
  -> 插件管理
  -> 通信机制

插件（Plugin）：
  -> 具体功能
  -> 独立开发
  -> 动态加载
```

**核心思想：**
- 核心系统稳定、小
- 功能以插件形式扩展
- 插件可独立开发和部署

### 微内核架构图

```
微内核（Core）
  -> 插件管理器
  -> 通信总线
  -> 生命周期管理

插件A（用户管理）
插件B（订单管理）
插件C（支付管理）
插件D（报表）
```

### 微内核的优缺点

**优点：**
- 核心稳定：核心小，变化少
- 灵活扩展：插件式扩展功能
- 隔离性好：插件间隔离
- 易于维护：插件独立开发

**缺点：**
- 设计复杂：需要设计插件接口
- 通信开销：插件间通信成本
- 版本管理：插件版本兼容

**适用场景：**
- 平台型产品
- 需要第三方扩展
- 功能变化频繁

## 六、架构模式对比

| 模式 | 拆分粒度 | 通信方式 | 数据库 | 适用场景 |
|------|---------|---------|--------|----------|
| 分层架构 | 层级拆分 | 函数调用 | 共享 | 中小型系统 |
| SOA | 服务拆分 | ESB路由 | 共享 | 企业集成 |
| 微服务 | 功能拆分 | 轻量通信 | 独立 | 互联网应用 |
| 微内核 | 插件拆分 | 总线通信 | 独立 | 平台产品 |

## 七、面试高频题

### 1. 分层架构和DDD分层有什么区别？
经典三层：展示、业务、数据 -> DDD分层：用户接口、应用、领域、基础设施 -> DDD强调领域层独立，不依赖技术实现

### 2. SOA和微服务的区别？
SOA：较粗粒度，ESB中心化，共享数据库 -> 微服务：较细粒度，去中心化，独立数据库 -> 微服务是SOA的演进

### 3. 微服务怎么拆分？
按业务能力拆分：用户、商品、订单 -> 按DDD限界上下文拆分 -> 三个火枪手原则：一个服务3个人负责 -> 避免过度拆分

### 4. 微内核架构适用于什么场景？
平台型产品：IDE、浏览器、ERP -> 需要第三方扩展 -> 功能变化频繁 -> 核心稳定，插件灵活
