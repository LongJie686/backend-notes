# 第4讲：高可用架构模式

## 核心结论（5条必记）

1. **CAP定理告诉我们不能三者兼得** -- 分区容错是必然，只能在一致性和可用性之间权衡，多数场景选择可用性优先
2. **FMEA是故障分析方法** -- 分析故障模式、影响、严重度，提前设计应对方案，避免线上被动
3. **存储高可用靠主备、主从、集群** -- 主备切换、主从复制、集群分区，不同场景选择不同方案
4. **计算高可用靠主备和集群** -- 主备热备、对称集群、非对称集群，关键是快速故障检测和切换
5. **异地多活是最终武器** -- 同城双活是基础，跨城异地是进阶，跨国异地是挑战，不要盲目追求

## 一、CAP定理

### CAP的含义

| 维度 | 说明 | 示例 |
|------|------|------|
| C (Consistency) | 一致性 | 所有节点同时看到相同数据 |
| A (Availability) | 可用性 | 每个请求都能得到响应（不保证是最新数据） |
| P (Partition Tolerance) | 分区容错性 | 网络分区时系统仍能运行 |

### 为什么P是必须的

**分布式系统必然有网络分区：**
- 网络抖动
- 交换机故障
- 机房断电
- 光纤挖断

**P是现实，C和A是选择：**
- CP：牺牲可用性，保证一致性
- AP：牺牲一致性，保证可用性
- CA：理论上存在，实际不适用

### CP vs AP

**CP场景（一致性优先）：**
- 银行转账：金额必须一致
- 库存扣减：不能超卖
- 配置中心：配置必须一致

**AP场景（可用性优先）：**
- 商品浏览：短时间数据不一致可接受
- 社交点赞：计数差异不影响体验
- 搜索：数据有延迟没关系

**权衡原则：**
- 强一致性：金融、支付
- 最终一致性：大多数互联网应用
- 弱一致性：缓存、搜索

## 二、FMEA分析方法

### FMEA是什么

FMEA（Failure Mode and Effects Analysis）= 故障模式与影响分析

**作用：**
- 识别可能的故障
- 分析故障影响
- 设计应对方案
- 提前准备，避免被动

### FMEA分析步骤

**1. 识别故障模式：**
```
系统：电商下单服务
故障模式：
  - 网络故障：无法连接数据库
  - 数据库故障：主库宕机
  - 依赖服务故障：库存服务不可用
  - 代码bug：逻辑错误
  - 流量激增：超出系统容量
```

**2. 分析故障影响：**
```
故障：数据库主库宕机
影响：
  - 无法创建订单
  - 无法查询订单
  - 影响范围：所有用户
  - 影响时长：主从切换30s
```

**3. 评估严重度：**
```
严重度 = 影响范围 × 影响时长 × 业务重要性

数据库主库宕机：
  - 影响范围：所有用户（高）
  - 影响时长：30s（中）
  - 业务重要性：下单是核心功能（高）
  -> 严重度：高
```

**4. 设计应对方案：**
``
故障：数据库主库宕机
应对：
  - 自动主从切换（30s内完成）
  - 限流降级（保护系统）
  - 监控告警（及时响应）
  - 容灾演练（定期演练）
```

### FMEA模板

| 故障模式 | 故障原因 | 影响范围 | 严重度 | 应对方案 |
|----------|----------|----------|--------|----------|
| 数据库主库宕机 | 硬件故障 | 全局 | 高 | 自动主从切换 |
| Redis集群不可用 | 网络分区 | 部分功能 | 中 | 降级走DB |
| 依赖服务超时 | 下游慢 | 局部 | 中 | 熔断降级 |
| 流量激增 | 活动推广 | 全局 | 高 | 限流扩容 |

## 三、高可用存储架构

### 主备复制

**架构：**
```
主库（Master）
  -> 同步复制到备库（Slave）
  -> 备库不提供服务
  -> 主库故障时人工切换到备库
```

**特点：**
- 备库只备份，不提供服务
- 切换需要人工介入
- 故障恢复时间长（分钟级）

**适用：**
- 对可靠性要求不高的场景
- 成本敏感的场景

### 主从复制

**架构：**
```
主库（Master）写入
  -> 异步复制到从库（Slave）
  -> 从库提供读服务
  -> 主库故障时切换到从库
```

**特点：**
- 读写分离，提升吞吐
- 主从切换（自动或半自动）
- 有数据延迟

**切换流程：**
```
1. 检测主库故障
2. 提升一个从库为主库
3. 其他从库重新同步到新主库
4. 更新应用连接配置
```

### 双主复制

**架构：**
```
主库A <-> 主库B（双向复制）
  -> 两个主库都提供读写服务
  -> 数据通过binlog互相同步
```

**优点：**
- 两个主库都可用
- 故障切换简单

**问题：**
- 数据冲突风险（同时修改同一条记录）
- 需要应用层规避冲突

**适用：**
- 两个机房异地部署
- 应用能规避数据冲突

### 集群与分区

**集群：**
```
多个节点组成集群
  -> 每个节点都提供服务
  -> 数据分片存储
  -> 一个节点故障不影响整体
```

**分区：**
```
数据按规则分片
  -> 每个节点存储一部分数据
  -> 查询时路由到对应节点
  -> 分区故障影响部分数据
```

## 四、高可用计算架构

### 主备计算

**架构：**
```
主服务器提供服务
  -> 备服务器 standby
  -> 主故障时切换到备
```

**切换方式：**
- 虚拟IP（VIP）：VIP在主备间漂移
- DNS切换：修改DNS解析
- 负载均衡摘除：从LB摘除故障节点

### 主从计算

**架构：**
```
主服务器提供服务
  -> 从服务器也提供服务
  -> 负载均衡分发流量
```

**与主备的区别：**
- 主从：两台都提供服务
- 主备：备机不提供服务，只 standby

### 对称集群

**架构：**
```
集群中所有节点对等
  -> 任何节点都能处理任何请求
  -> 无主从之分
```

**优点：**
- 节点对等，管理简单
- 扩容方便

**缺点：**
- 需要数据无状态
- 需要分布式锁机制

**代表：** Cassandra、Elasticsearch

### 非对称集群

**架构：**
```
集群中节点角色不同
  -> Master节点负责管理
  -> Worker节点负责计算
  -> 有主从之分
```

**优点：**
- 职责明确
- 便于管理

**缺点：**
- Master是瓶颈
- Master故障需要选举

**代表：** Hadoop（NameNode + DataNode）

## 五、接口级故障应对

### 降级

**什么是降级：**
牺牲非核心功能，保障核心功能可用

**降级策略：**

| 策略 | 说明 | 示例 |
|------|------|------|
| 返回默认值 | 返回空或默认数据 | 推荐列表为空 |
| 返回缓存 | 返回过期缓存数据 | 商品详情缓存 |
| 关闭非核心功能 | 直接关闭 | 评论、点赞 |
| 静默失败 | 不处理，直接返回 | 统计日志 |

**降级开关：**
```java
@GetMapping("/product/detail")
public Product getProductDetail(Long productId) {
    Product product = productService.getProduct(productId);

    // 推荐功能降级
    if (!FeatureToggle.isOn("recommend")) {
        product.setRecommendations(Collections.emptyList());
    }

    // 评论功能降级
    if (!FeatureToggle.isOn("comment")) {
        product.setComments(Collections.emptyList());
    }

    return product;
}
```

### 熔断

**熔断器状态机：**
```
关闭 -> 打开 -> 半开 -> 关闭
正常   熔断   探测   恢复
```

**熔断触发条件：**
- 异常率超过阈值（如50%）
- 响应时间超过阈值（如1s）
- 持续一定时间（如10s）

### 限流

**限流维度：**
```
全局限流：保护整体系统
接口限流：保护单个接口
用户限流：防止单用户刷接口
IP限流：防止恶意攻击
```

**限流算法：**
- 令牌桶：允许突发流量
- 漏桶：强制平滑流量
- 滑动窗口：精确限流

## 六、异地多活

### 为什么需要异地多活

**单机房风险：**
- 机房断电、网络故障
- 自然灾害（地震、火灾）
- 市政施工挖断光纤

**异地多活价值：**
- 故障隔离：一个机房故障不影响整体
- 就近访问：用户访问最近机房
- 容灾备份：数据异地备份

### 同城双活

**架构：**
```
同城两个机房
  -> 两个机房同时提供服务
  -> 数据实时同步
  -> 一个机房故障，流量切到另一个
```

**特点：**
- 距离近（10-50km）
- 延迟低（<5ms）
- 成本相对低

**挑战：**
- 无法应对城市级灾难
- 跨机房专线成本

### 跨城异地

**架构：**
```
两个不同城市
  -> 北京 + 上海
  -> 距离远（1000+ km）
  -> 延迟高（20-50ms）
```

**特点：**
- 能应对城市级灾难
- 延迟较高，需要应用配合

**挑战：**
- 数据一致性：两机房数据同步
- 路由策略：用户路由到哪个机房
- 运维复杂：需要两个团队

### 异地多活架构设计

**4大技巧：**
1. **数据分层**：核心数据强一致，非核心数据最终一致
2. **单元化**：按用户ID划分，一个用户固定在一个单元
3. **路由规则**：用户请求路由到归属单元
4. **数据同步**：异步同步非核心数据

**4个步骤：**
1. **业务分级**：核心业务、重要业务、一般业务
2. **数据分类**：强一致数据、最终一致数据
3. **架构设计**：同步方案、路由方案、降级方案
4. **灰度验证**：小流量验证，逐步放大

## 七、面试高频题

### 1. CAP定理是什么？
分布式系统不能同时满足一致性、可用性、分区容错性 -> 分区容错是必然 -> 实际选择CP或AP -> 大多数互联网应用选AP（可用性优先）

### 2. FMEA分析是什么？
故障模式与影响分析 -> 识别故障、分析影响、评估严重度、设计方案 -> 提前准备避免线上被动

### 3. 主备和主从有什么区别？
主备：备机不提供服务，只standby -> 主从：两台都提供服务，一主一从或双主 -> 主从切换更复杂但资源利用率高

### 4. 异地多活有哪些挑战？
数据一致性：两机房数据同步延迟 -> 路由策略：用户路由到哪个机房 -> 运维复杂：需要两套团队 -> 成本高：双倍资源
