# 第3讲：RAG 系统设计与实现

> 阶段目标：从 0 到 1 构建一个生产级 RAG 系统。

## 学习目标

- 掌握 RAG 完整链路
- 理解文档切分策略与选型
- 掌握向量数据库使用
- 实现混合检索与重排序

## 核心内容

### RAG 完整流程

```
文档加载 -> 文本切分 -> Embedding -> 向量存储 -> 检索 -> 重排序 -> 生成
```

**为什么需要 RAG：**
- 大模型的知识有截止日期
- 企业私有知识模型没见过
- 直接让模型回答会产生幻觉
- RAG 把相关知识"喂"给模型，让回答有据可依

### 文档加载

| 格式 | 工具 | 说明 |
|------|------|------|
| PDF | PyPDF2 / pdfplumber | 处理表格和图片需要 OCR |
| Word | python-docx | 按段落提取 |
| Markdown | 直接读取 | 天然分段，最友好 |
| HTML | BeautifulSoup | 需要去标签 |
| 数据库 | SQLAlchemy | 结构化数据转文本 |

### 文档切分策略

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 固定长度 | 按字符数切 | 简单 | 可能切断语义 |
| 递归切分 | 按分隔符层级切 | 保留段落结构 | 需要调参 |
| 语义切分 | 按 Embedding 相似度切 | 语义完整 | 计算成本高 |
| 按标题切分 | 按文档标题层级 | 结构清晰 | 依赖文档结构 |

**实用经验：**
- chunk_size 一般 500-1000 字符
- chunk_overlap 建议 chunk_size 的 10%-20%
- 不同文档类型用不同策略

### Embedding 模型选型

| 模型 | 维度 | 特点 | 适用场景 |
|------|------|------|----------|
| OpenAI text-embedding-3 | 1536/3072 | 效果好，付费 | 通用 |
| BGE-large-zh | 1024 | 中文效果好，开源 | 中文场景 |
| M3E-base | 768 | 中文多任务，开源 | 轻量中文 |
| Jina-embeddings-v2 | 1024 | 支持长文本 | 长文档 |

### 向量数据库

| 数据库 | 类型 | 特点 |
|--------|------|------|
| FAISS | 内存库 | 轻量，适合原型 |
| Chroma | 嵌入式 | 开发友好 |
| Milvus | 分布式 | 生产级，高性能 |
| Qdrant | 分布式 | Rust 实现，高效 |
| Pinecone | 云服务 | 免运维 |

### 检索策略

**稠密检索（Dense Retrieval）：**
- 用 Embedding 向量的相似度检索
- 语义匹配能力强
- 适合模糊查询

**稀疏检索（BM25）：**
- 基于词频的统计检索
- 精确匹配能力强
- 适合关键词明确的查询

**混合检索（Hybrid Search）：**

```
最终得分 = alpha * 稠密检索得分 + (1-alpha) * 稀疏检索得分
```

- 结合语义匹配和精确匹配
- alpha 通常 0.5-0.7
- 效果通常优于单一检索

### 重排序（Reranking）

**为什么需要：**
- 初始检索是"双塔"结构，粗筛快但不够精确
- 重排序用 Cross-Encoder 做精细打分
- 检索 Top-K（如 20），重排序取 Top-N（如 5）

**常用方案：**
- Cohere Rerank API
- BGE-Reranker
- bce-reranker-base

### 高级 RAG 技术

| 技术 | 原理 | 解决什么问题 |
|------|------|------------|
| 查询改写 | 用 LLM 改写用户查询 | 查询太模糊或不精确 |
| HyDE | 用 LLM 生成假设答案来检索 | 查询和文档语义鸿沟大 |
| Self-RAG | 让模型自己判断是否需要检索 | 避免不必要的检索 |
| 知识图谱增强 | 用知识图谱辅助检索 | 需要关系推理的场景 |
| 多路召回 | 多种检索方式结果融合 | 提高召回率 |

## 重点认知

1. **文档切分是 RAG 效果的基础**，切不好后面全白搭
2. **混合检索通常比纯向量检索效果好**
3. **重排序是性价比最高的优化手段**
4. **RAG 不是"扔进去就行"**，需要精细调优

## 实战建议

1. 构建一个企业知识库问答系统：
   - 接入公司文档（PDF/Markdown）
   - 实现文档切分 + Embedding + 向量存储
   - 对比不同切分策略的效果
   - 实现混合检索 + 重排序
2. 量化评估：准确率、召回率、回答相关性

## 常见问题

**Q: 切分粒度怎么选？**
A: 切太细丢失上下文，切太粗检索不精确。一般 500-1000 字符，根据文档类型调整。

**Q: 为什么检索到的内容相关但答案不对？**
A: 可能是生成阶段的 Prompt 没有引导模型"基于以下信息回答"，也可能是检索结果太多导致模型被干扰。

**Q: RAG 和微调怎么选？**
A: 需要实时更新知识用 RAG，需要改变模型行为风格用微调，两者可以结合。
