# 面试高频问题

按类别整理大模型应用开发方向的面试高频问题，帮助你有针对性地准备。

## 基础原理类

### Q1：Transformer 的自注意力机制是什么？为什么比 RNN 更好？

自注意力机制允许模型在处理每个 Token 时关注输入序列中的所有位置，通过 Query-Key-Value 三组矩阵计算注意力权重。相比 RNN 的优势：(1) 支持并行计算，训练速度快；(2) 直接建模长距离依赖，不受序列长度限制；(3) 通过多头注意力可以捕获不同类型的关系。

### Q2：GPT 和 BERT 的核心区别是什么？

GPT 是自回归模型，从左到右逐个生成 Token，适合生成任务；BERT 是掩码语言模型，通过预测被遮盖的 Token 来训练，适合理解任务（分类、NER）。GPT 使用 Decoder-only 架构，BERT 使用 Encoder-only 架构。

### Q3：什么是 Token？为什么同样的文字在不同模型中 Token 数不同？

Token 是模型处理文本的最小单位。不同模型使用不同的 Tokenizer（BPE、WordPiece、SentencePiece），训练数据也不同，所以对同一文本的分词结果不同。中文在 GPT 系列中通常一个字 1-2 个 Token，在专用中文模型中可能更少。

### Q4：Temperature、Top-K、Top-P 分别怎么影响生成结果？

Temperature 控制概率分布的平滑度，越低越确定；Top-K 限制只从概率最高的 K 个 Token 中采样；Top-P 从累积概率超过 P 的最小 Token 集合中采样。三者通常配合使用：先设 Temperature，再用 Top-P 或 Top-K 做截断。

### Q5：什么是上下文窗口？它对应用开发有什么影响？

上下文窗口是模型单次能处理的最大 Token 数量。它限制了单次请求中系统提示词 + 对话历史 + 检索结果 + 用户输入 + 模型输出的总长度。影响包括：对话管理需要策略来裁剪历史，RAG 系统需要控制检索结果数量，长文档处理需要切分。

## Prompt 类

### Q6：什么是 Zero-shot、Few-shot、Chain-of-Thought？

Zero-shot 直接给任务描述不给示例；Few-shot 提供几个输入输出示例让模型学习模式；Chain-of-Thought 引导模型展示推理过程。CoT 对数学、逻辑推理任务效果显著提升，但会增加 Token 消耗。

### Q7：如何控制模型的输出格式？

几种方法：(1) 在 Prompt 中明确指定格式要求；(2) 使用 Few-shot 示范期望格式；(3) 在输出开头给出引导（如直接输出 "{ "）；(4) 使用 API 的 response_format 参数强制 JSON 输出；(5) 后处理用正则或代码提取结构化内容。

### Q8：Prompt 注入是什么？如何防御？

Prompt 注入是攻击者通过用户输入或外部数据植入恶意指令，改变模型行为。防御方法：(1) 输入预处理过滤可疑模式；(2) 用分隔符明确隔离用户输入；(3) 系统提示词中强调安全规则；(4) 输出审核检测异常行为；(5) 最小权限原则限制工具访问。

### Q9：如何评估 Prompt 的效果？

建立评估数据集（标注好的输入-期望输出对），从多个维度量化评估：准确率、格式符合率、相关性、安全性。每次修改 Prompt 前后跑评估对比，用自动化工具（如 Promptfoo）做回归测试。建议至少 50-100 条评估数据。

## RAG 类

### Q10：RAG 的完整流程是什么？

离线阶段：文档收集 -> 解析 -> 切分 -> 向量化 -> 存入向量数据库。在线阶段：用户查询 -> 查询处理（改写/扩展） -> 向量检索 -> 重排序 -> 上下文组装 -> LLM 生成回答。

### Q11：文档切分有哪些策略？chunk_size 设多大合适？

常见策略：固定长度切分、按段落/章节切分、递归切分。chunk_size 建议 300-500 用于精确检索，800-1000 用于需要更多上下文的场景。关键原则：保证语义完整性，设置 10%-20% 的重叠。

### Q12：向量数据库怎么选？

开发阶段用 Chroma/FAISS 快速验证；生产环境根据数据量选择：百万级以内 Milvus/Qdrant 单节点即可，亿级需要分布式方案。选型考虑：数据规模、查询性能、是否需要混合搜索、运维成本。

### Q13：混合检索是什么？为什么比单一检索好？

混合检索结合稠密检索（向量相似度）和稀疏检索（BM25 关键词匹配）。稠密检索理解语义但可能遗漏关键词精确匹配；稀疏检索精确匹配但不理解语义。两者互补，混合检索通常比单一检索效果好 10-20%。

### Q14：重排序（Reranking）是什么？为什么需要？

初始检索使用双塔模型（查询和文档独立编码），速度快但精度有限。重排序使用交叉编码器（同时处理查询和文档），精度更高但速度慢。两阶段策略：快速检索缩小范围，精细模型排序，用少量计算换取更高的准确性。

## Agent 类

### Q15：Agent 和普通的 LLM 调用有什么区别？

普通调用是单轮的 Prompt-Response 模式；Agent 具有自主决策能力，可以多轮迭代、调用工具、维护状态。Agent = LLM + 工具 + 记忆 + 规划，能够自主完成复杂的多步骤任务。

### Q16：Function Calling 的工作原理是什么？

开发者定义可用函数的名称、描述和参数格式；模型判断是否需要调用函数，如果需要则返回函数名和参数；开发者执行函数并将结果返回给模型；模型基于函数结果生成最终回答。模型本身不执行函数，只决定调用什么。

### Q17：ReAct 范式是什么？

ReAct（Reasoning + Acting）让模型交替进行推理（Thought）和行动（Action），每一步先思考当前状态，再决定下一步行动，然后观察结果。循环直到达成目标。优势是推理过程可解释，劣势是 Token 消耗大。

### Q18：如何防止 Agent 陷入无限循环？

(1) 设置最大迭代次数（5-10次）；(2) 记录已调用的工具和参数，检测重复调用；(3) 在 Prompt 中明确定义完成条件和停止时机；(4) 实现全局超时机制；(5) 监控每次迭代的结果变化，如果没有新进展则强制终止。

### Q19：多 Agent 架构有哪些常见模式？

三种常见模式：(1) 主管-工人：一个主管 Agent 分配任务给多个工人 Agent；(2) 流水线：多个 Agent 按顺序处理，前一个的输出是后一个的输入；(3) 辩论：多个 Agent 从不同角度分析，最后由评委 Agent 综合结论。

## 工程化类

### Q20：大模型应用的流式响应怎么实现？

服务端使用 SSE（Server-Sent Events）协议：API 调用模型时设置 stream=True，将模型逐个生成的 Token 通过 HTTP 流式返回。前端使用 fetch + ReadableStream 或 EventSource 接收。注意事项：关闭 Nginx 缓冲（proxy_buffering off），设置合理的超时时间。

### Q21：如何控制大模型应用的成本？

多管齐下：(1) 根据任务复杂度选择合适的模型（简单任务用小模型）；(2) 实现缓存策略，重复查询返回缓存结果；(3) 控制 Prompt 长度，减少不必要的上下文；(4) 设置输出 max_tokens 限制；(5) 实现用户限流和预算管理；(6) 监控 Token 消耗趋势，异常告警。

### Q22：vLLM、TGI、Ollama 分别适合什么场景？

vLLM 适合高并发生产环境，吞吐量最高；TGI 适合 HuggingFace 生态用户；Ollama 适合开发和小规模部署，支持 CPU。选择建议：开发用 Ollama，生产用 vLLM 或 TGI，没有 GPU 直接调 API。

### Q23：大模型应用需要监控哪些指标？

性能指标：响应时间（P50/P95/P99）、首 Token 时间、生成速度（Tokens/s）；可用性指标：错误率、API 限流率；成本指标：Token 消耗量、单次请求成本；业务指标：用户满意度、回答准确率；资源指标：GPU 利用率、内存使用率。

## 微调类

### Q24：什么时候该用微调，什么时候该用 RAG？

RAG 适合需要外部知识的场景，知识可以随时更新。微调适合需要改变模型行为（风格、格式、推理模式）的场景。决策流程：先尝试 Prompt 工程 -> 效果不够加 RAG -> 还不够考虑微调。大多数场景 Prompt + RAG 就够了。

### Q25：LoRA 的原理是什么？为什么比全量微调好？

LoRA 不修改原始模型参数，而是在旁边添加低秩矩阵（B*A）来学习适配。参数量通常只有原始模型的 0.1%-1%。优势：训练快、显存低、不修改原始模型、可以随时切换不同的 LoRA 适配器、推理时可以合并到原始权重无额外开销。

### Q26：QLoRA 和 LoRA 有什么区别？

QLoRA 在 LoRA 基础上先将模型量化为 4-bit，然后在量化模型上训练 LoRA。这使得 7B 模型可以在单张 RTX 3090（24GB）上微调，而普通 LoRA 需要约 16GB 显存。效果损失很小，性价比极高。

### Q27：微调需要多少数据？数据质量有多重要？

数据质量远比数量重要。风格迁移最少 500 条（推荐 2000-5000 条），领域知识至少 2000 条（推荐 5000-20000 条）。关键标准：输出正确、格式统一、场景多样、无噪音。1000 条高质量数据比 10000 条低质量数据效果好。
