# 常见坑点总结

在大模型应用开发中，有一些反复出现的典型错误。这里总结了 10 个最常见的坑，帮助你在项目中避坑。

## 坑点 1：把大模型当数据库用

### 问题表现

把大量数据塞进 Prompt，让模型"记住"并回答。比如把整个产品目录放进上下文，让用户直接问"有哪些红色手机"。

### 为什么是坑

- 上下文窗口有限（即使 128K 也不够存大量数据）
- 模型不擅长精确检索（可能遗漏或编造数据）
- Token 成本极高（每次请求都要发送全部数据）
- 数据更新需要重新发送（模型不会"记住"上一次的对话）

### 正确做法

- 使用 RAG 系统：数据存向量数据库，检索相关片段后传给模型
- 使用结构化数据库：精确查询用 SQL，模型只做理解和生成
- 分层设计：数据库负责精确查询，模型负责理解和总结

## 坑点 2：Prompt 写得像散文

### 问题表现

Prompt 是一大段自然语言描述，没有结构、没有重点、没有约束。模型每次理解都不一样，输出不稳定。

### 为什么是坑

- 模型的注意力有限，长文本中关键信息容易被忽略
- 缺乏结构导致模型自由发挥，输出格式不可控
- 迭代优化困难，不知道改哪里有效果

### 正确做法

- 使用结构化格式：角色-任务-上下文-格式-约束
- 用标题和分隔符组织信息
- 明确指定输出格式和长度
- 用 Few-shot 示范期望的输入输出

## 坑点 3：RAG 系统"检索了但没用好"

### 问题表现

RAG 系统能检索到相关文档，但最终回答质量依然很差。检索结果要么太长塞不下，要么太短信息不足，或者被无关信息干扰。

### 为什么是坑

- 检索到不等于用好，上下文组装的方式影响巨大
- 检索结果可能包含冲突信息，模型不知道信哪个
- 没有引导模型如何使用检索结果

### 正确做法

- 检索后做重排序（Reranking），提升相关性
- 控制注入的检索结果数量（通常 3-5 条）
- 在 Prompt 中明确指示如何使用检索结果
- 对检索结果做摘要或提取关键信息后再注入

## 坑点 4：Agent 无限循环不收敛

### 问题表现

Agent 反复调用相同的工具，陷入循环无法得出结论。或者每次思考都说"我需要更多信息"，永远不给出最终答案。

### 为什么是坑

- Agent 的停止条件不明确
- 模型可能对同一个问题反复思考
- 工具返回的结果不符合预期，导致重复调用

### 正确做法

- 设置最大迭代次数（通常 5-10 次）
- 记录已调用的工具和参数，避免重复调用
- 明确定义完成条件，在 Prompt 中强调何时应该停止
- 实现超时机制，强制终止长时间运行的任务

## 坑点 5：忽视流式输出体验

### 问题表现

大模型生成一段 500 字的回答可能需要 10-20 秒，如果等全部生成完再返回，用户面对的是长时间的白屏。

### 为什么是坑

- 用户体验极差，感觉系统"卡住了"
- 用户不知道系统在处理，可能重复提交
- 对于长文本生成，等待时间成倍增加

### 正确做法

- 所有文本生成都使用流式响应（SSE）
- 显示"正在思考"状态，首 Token 时间控制在 2 秒以内
- 前端逐字渲染，让用户实时看到生成过程
- 设置合理的超时时间，避免无限等待

## 坑点 6：没有做敏感内容过滤

### 问题表现

直接将用户输入传给模型，模型输出直接返回给用户。没有内容审核，可能产生有害、违法或不恰当的内容。

### 为什么是坑

- 法律风险：可能违反内容审核相关规定
- 品牌风险：不当输出影响产品口碑
- 用户风险：对特定人群（如未成年人）造成伤害
- 业务风险：竞品或恶意用户利用漏洞

### 正确做法

- 输入层做关键词过滤和意图检测
- 输出层做内容审核（关键词 + 分类模型）
- 系统提示词中明确安全边界
- 建立人工审核机制处理灰色地带

## 坑点 7：Token 成本不控制

### 问题表现

没有监控 Token 消耗，没有预算控制，没有缓存策略。月底一看账单，API 费用远超预期。

### 为什么是坑

- 大模型 API 按使用量计费，成本随用户量线性增长
- 一个复杂查询可能消耗数千 Token
- 没有监控就无法发现异常消耗
- 没有预算控制，恶意用户可以刷爆账单

### 正确做法

- 实现请求级别的 Token 计量和记录
- 设置每日/每月消费预算和告警
- 实现缓存策略，重复查询直接返回缓存
- 根据任务复杂度选择合适的模型（简单任务用小模型）
- 对用户做限流，防止单用户过度消耗

## 坑点 8：把所有问题都交给一个大模型

### 问题表现

用一个通用大模型处理所有任务：分类、提取、翻译、总结、推理。不考虑每个任务的最优解。

### 为什么是坑

- 简单任务用大模型成本高、延迟大
- 复杂任务小模型做不好
- 不同任务的最佳参数不同，无法用一套配置覆盖

### 正确做法

- 分级使用：简单任务用小模型/规则引擎，复杂任务用大模型
- 专业化：特定任务用微调模型或专用模型（如情感分析用专用模型）
- 组合使用：规则引擎做预处理，大模型做理解和生成
- 建立模型路由，根据任务类型自动选择最优模型

## 坑点 9：没有评估体系就上线

### 问题表现

凭主观感觉判断效果好不好，没有量化指标，没有 A/B 测试，没有回归测试。上线后效果下降也不知道。

### 为什么是坑

- 无法量化改进是否有效
- 修改一处可能影响另一处（回归问题）
- 没有基线数据，无法判断当前效果是否达标
- 团队对效果没有共识

### 正确做法

- 上线前建立评估数据集（标注好的输入-期望输出对）
- 定义核心指标：准确率、相关性、安全性、延迟
- 每次改动前后跑评估，量化效果变化
- 实现自动化评估流水线
- 上线后持续监控指标，发现异常及时处理

## 坑点 10：架构一开始就设计成单体

### 问题表现

把 Prompt 管理、API 调用、缓存、日志、内容审核等所有逻辑写在一个服务里。初期开发快，后期维护困难。

### 为什么是坑

- 改 Prompt 需要重新部署整个服务
- 换模型需要改大量代码
- 无法独立扩缩容
- 一个组件出问题影响整体
- 团队协作困难

### 正确做法

- 模块化设计：Prompt 管理、模型调用、缓存、审核各自独立
- 通过接口解耦，方便替换底层实现
- Prompt 模板外置到配置文件或数据库
- 模型调用层抽象统一接口，支持多模型切换
- 核心服务（API 网关）和非核心服务（日志、审核）分离部署
