# 第5讲：多轮对话与情感机器人项目

## 核心结论（5 条必记）

1. **多轮对话的核心是上下文管理** -- 不是简单地把所有历史都塞给模型，需要滑动窗口、摘要替换、关键信息提取等策略
2. **会话状态机驱动对话流转** -- 每个状态有明确的进入和退出条件，状态转换由意图识别驱动，异常状态有兜底处理
3. **情感分析不需要 100% 准确** -- 关键是趋势判断和异常检测，用情感维度（积极度、激动度、具体情绪、强度）描述情感状态
4. **安全过滤是底线** -- 输入输出双层过滤，分级处理（高危拦截、中危引导、低危放行但记录）
5. **个性化要循序渐进** -- 不要一上来就过度推断，根据交互逐步积累用户画像

## 一、多轮对话上下文管理

**核心挑战：** 上下文窗口有限，对话历史会越来越长。

**管理策略：**

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 滑动窗口 | 只保留最近 N 轮对话 | 通用场景 |
| 摘要替换 | 用 LLM 对历史对话做摘要 | 长对话 |
| 重要信息提取 | 从对话中提取关键实体和意图 | 业务对话 |
| 混合策略 | 近期对话 + 历史摘要 + 关键信息 | 生产系统 |

**实现示例：**

```python
class ConversationManager:
    def __init__(self, max_turns=10):
        self.max_turns = max_turns
        self.summary = ""
        self.key_entities = {}

    def add_message(self, role, content):
        self.messages.append({"role": role, "content": content})
        if len(self.messages) > self.max_turns:
            self._summarize_old_messages()

    def get_context(self):
        return [
            {"role": "system", "content": f"对话摘要：{self.summary}"},
            *self.messages[-self.max_turns:]
        ]
```

## 二、会话状态机与情感分析

### 会话状态机设计

**状态定义：**

```
GREETING -> COLLECTING_INFO -> PROCESSING -> RESPONDING -> CLOSING
     ^                                              |
     |______________________________________________|
```

**关键设计：**
- 每个状态有明确的进入和退出条件
- 状态转换由意图识别驱动
- 异常状态有兜底处理
- 支持从任意状态回到初始状态

### 情感分析与识别

**集成方案：**

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| LLM 直接分析 | 在 Prompt 中要求分析情感 | 灵活但成本高 |
| 专用模型 | 用 BERT 等模型做情感分类 | 快但不够灵活 |
| 规则 + 模型 | 关键词匹配 + 模型补充 | 平衡方案 |

**情感维度：**

```
{
    "valence": 0.8,      # 积极度 -1 到 1
    "arousal": 0.3,      # 激动度 0 到 1
    "emotion": "happy",  # 具体情绪
    "intensity": 0.6     # 强度 0 到 1
}
```

## 三、敏感内容检测与用户画像

### 敏感内容检测与过滤

**多层过滤架构：**

```
用户输入
  -> 关键词过滤（快速）
  -> 分类模型（中等）
  -> LLM 审核（精确）
  -> 业务规则（兜底）
生成输出
  -> 同样的过滤流程
```

**实现要点：**
- 敏感词库定期更新
- 分类模型要覆盖主要风险类别
- 过滤不能影响正常对话体验
- 记录所有触发过滤的案例用于优化

### 用户画像与个性化

**画像维度：**
- 基本信息：年龄、性别、职业
- 兴趣偏好：话题偏好、沟通风格
- 情感状态：当前情绪、情绪变化趋势
- 交互模式：活跃时间、对话频率

**个性化策略：**
- 根据情感状态调整回复语气
- 根据兴趣偏好推荐话题
- 根据交互模式优化回复时机

## 四、项目实战："心语"情感机器人

### 系统架构

```
前端（Web/小程序）
  -> API 网关
    -> 对话管理服务
      -> 情感分析模块
      -> RAG 知识库
      -> LLM 服务
      -> 安全审核模块
    -> 用户服务
    -> 日志服务
```

### 核心模块

1. **对话管理模块**：多轮上下文、状态流转
2. **情感分析模块**：实时情感识别、情绪追踪
3. **知识库模块**：RAG 检索情感支持知识
4. **安全审核模块**：输入输出双层过滤
5. **个性化模块**：用户画像、个性化回复

### API 设计

```
POST /api/chat          -- 发送消息
GET  /api/history       -- 获取对话历史
POST /api/feedback      -- 提交用户反馈
GET  /api/user/profile  -- 获取用户画像
```

## 五、面试高频题

### 1. 多轮对话的上下文窗口满了怎么办？
对话历史超过上下文窗口限制 -> 用摘要替换策略，把早期对话压缩成摘要 -> 保留近期原始对话 + 历史摘要 + 关键信息

### 2. 如何在对话中保持一致的"人设"？
System Prompt 定义角色但多轮对话中容易偏移 -> 在每次对话中都携带 System Prompt，关键信息从画像中提取加入 -> 结合状态机保证行为一致性

### 3. 敏感内容过滤怎么做到不影响正常对话体验？
一次性拦截会误杀 -> 分级处理：高危内容直接拦截，中危内容柔性引导，低危内容放行但记录 -> 持续优化过滤规则
