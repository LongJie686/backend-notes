# 第4讲：分库分表

## 核心结论（5条必记）

1. **何时分库分表有判断标准** -- 单表数据>1000万、单库数据>5000万、数据库QPS>单机容量，才考虑拆分
2. **分片键选择至关重要** -- 决定数据分布和查询性能，大多数查询能通过分片键定位
3. **垂直拆分按业务，水平拆分按数据** -- 垂直拆分解决表过多问题，水平拆分解决数据量大问题
4. **分库分表后分布式事务困难** -- 尽量避免跨库事务，通过消息队列实现最终一致性
5. **数据迁移要在线进行** -- 双写+数据同步+灰度切换，保证业务不中断

---

## 一、什么时候需要分库分表

### 判断标准

| 指标 | 阈值 | 说明 |
|------|------|------|
| 单表数据量 | >1000万 | 查询慢、索引大 |
| 单库数据量 | >5000万 | 备份慢、恢复慢 |
| 数据库QPS | >单机容量（1-2万） | CPU、IO打满 |

### 优先级

**先做这些优化：**
```
1. SQL优化
2. 索引优化
3. 缓存优化
4. 读写分离
```

**都不行了才分库分表：**
```
5. 垂直拆分（按业务）
6. 水平拆分（按数据）
```

---

## 二、垂直拆分

### 拆分原则

**按业务拆分：**
```
单体数据库
  -> 用户库（user、profile、account）
  -> 商品库（product、category、inventory）
  -> 订单库（order、order_item、payment）
```

**好处：**
- 单库压力减少
- 业务清晰
- 团队并行开发

---

## 三、水平拆分

### 拆分策略

**按范围拆分：**
```
user_id 0-100万     -> DB1
user_id 100-200万   -> DB2
user_id 200-300万   -> DB3
```

**按Hash拆分：**
```
DB = hash(user_id) % 分库数
Table = hash(user_id) / 分库数 % 分表数
```

**按地理位置：**
```
华北 -> DB1
华东 -> DB2
华南 -> DB3
```

---

## 四、分片键选择

### 选择原则

1. **查询时能定位分片**：大多数查询包含分片键
2. **数据分布均匀**：避免热点
3. **避免跨片查询**：减少性能损失

### 示例

| 表 | 分片键 | 原因 |
|----|--------|------|
| user | user_id | 按用户查询 |
| order | user_id | 大多数查询按用户查订单 |
| product | product_id | 按商品查询 |

---

## 五、分布式事务

### 本地消息表

**流程：**
```
Service A：
  1. 开启事务
  2. 执行业务
  3. 插入消息表
  4. 提交事务

定时任务：
  1. 扫描消息表
  2. 调用Service B
  3. 成功则删除消息
```

## 六、面试高频题

### 1. 分库分表怎么拆？
垂直拆分：按业务拆分 -> 水平拆分：按数据量拆分 -> 垂直先做，水平后做

### 2. 分片键怎么选？
大多数查询能通过分片键定位 -> 数据分布均匀 -> 避免跨片查询

### 3. 分库分表后怎么查询？
通过分片键定位 -> 分片查询后聚合 -> 禁止跳页 -> 搜索引擎
