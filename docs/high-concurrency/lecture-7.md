# 第7讲：场景实战（秒杀、Feed流、计数系统）

## 核心结论（5条必记）

1. **秒杀系统核心是防超卖** -- Redis原子扣减、限流拦截、异步下单，确保不超卖
2. **Feed流系统关键是存储优化** -- 写扩散、读扩散、混合模式，按用户量级选择
3. **计数系统要保证准确性** -- Redis计数+持久化、定期归档、分片存储
4. **场景设计要权衡取舍** -- 没有完美方案，根据业务阶段和团队能力选择
5. **持续优化和演进** -- 从简单到复杂，随着业务增长逐步优化

## 一、秒杀系统

### 核心挑战

```
1. 高并发：瞬间流量暴增（100万QPS）
2. 防超卖：库存不能超卖
3. 防刷：脚本机器人刷单
```

### 架构设计

```
用户 -> CDN -> 限流 -> 秒杀服务 -> Redis（扣库存）
                         -> MQ（异步下单）
                         -> 订单服务
```

### 核心代码

```java
@Service
public class SeckillService {

    @Autowired
    private RedisTemplate redisTemplate;

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public Result seckill(Long userId, Long productId) {
        // 1. 限流
        if (!rateLimiter.tryAcquire()) {
            return Result.error("当前人数过多");
        }

        // 2. 校验库存
        String stockKey = "stock:" + productId;
        Long stock = redisTemplate.opsForValue().decrement(stockKey);

        if (stock == null || stock < 0) {
            redisTemplate.opsForValue().increment(stockKey); // 回滚
            return Result.error("库存不足");
        }

        // 3. 异步下单
        rocketMQTemplate.asyncSend("order-topic", new Order(userId, productId));

        return Result.success("抢购成功");
    }
}
```

## 二、Feed流系统

### 核心挑战

```
1. 高并发：大量读写请求
2. 数据量大：存储压力大
3. 排序：按时间、权重排序
```

### 存储方案

**写扩散：**
```
用户发推：
  -> 写入所有粉丝的收件箱
优点：读快
缺点：写慢，大V不适用
```

**读扩散：**
```
用户读Feed：
  -> 拉取关注的人的推
优点：写快
缺点：读慢
```

**混合模式：**
```
大V：写扩散
普通用户：读扩散
```

## 三、计数系统

### 核心挑战

```
1. 高并发：大量计数请求
2. 准确性：计数不能错
3. 性能：响应要快
```

### 设计方案

```
Redis计数（实时）：
  -> INCR key
  -> 性能好
  -> 可能丢失

异步持久化：
  -> 定时批量写入DB
  -> 保证最终一致
```

### 分片存储

```
按用户ID分片：
  -> count:0-100万
  -> count:100-200万
  -> count:200-300万
```

## 四、面试题总结

### 秒杀系统

**问：** 怎么防止超卖？
**答：** Redis原子扣减，Lua脚本保证原子性

**问：** 怎么防止脚本刷单？
**答：** 限流、验证码、IP限制

### Feed流系统

**问：** 怎么优化大V场景？
**答：** 写扩散，大V的推提前写入粉丝收件箱

### 计数系统

**问：** 怎么保证计数准确？
**答：** Redis计数 + 异步持久化 + 定期对账
