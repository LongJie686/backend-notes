# 第6讲：分布式数据一致性

## 核心结论（5条必记）

1. **CAP定理告诉我们不能三者兼得** -- 一致性、可用性、分区容错性只能满足两个，大多数场景选择AP（可用性优先）
2. **BASE理论是最终一致性** -- 基本可用、软状态、最终一致性，牺牲强一致性换取高可用
3. **分布式事务没有银弹** -- 2PC强一致但性能差，TCC业务侵入高，Saga适合长流程，本地消息表最实用
4. **幂等性是分布式系统的基石** -- 没有幂等的重试会导致重复消费，通过唯一ID、状态机、唯一索引保证幂等
5. **补偿机制和对账是兜底** -- 技术手段保证不了100%一致，需要业务补偿和对账机制处理不一致数据

## 一、CAP定理

### CAP的含义

| 维度 | 说明 | 示例 |
|------|------|------|
| C (Consistency) | 一致性 | 所有节点同时看到相同数据 |
| A (Availability) | 可用性 | 每个请求都能得到响应 |
| P (Partition Tolerance) | 分区容错性 | 系统在网络分区时仍能运行 |

### CAP的权衡

**理论上只能满足两个：**

| 组合 | 说明 | 适用场景 |
|------|------|----------|
| CA | 强一致、高可用，不允许分区 | 单机系统，分布式不适用 |
| CP | 强一致、容分区，牺牲可用性 | 金融系统，强一致性要求 |
| AP | 高可用、容分区，牺牲一致性 | 互联网应用，多数场景 |

**网络分区必然存在：**
- 分布式系统中，网络故障不可避免
- P是必须满足的
- 实际选择是CP还是AP

### CP vs AP

**CP场景：**
- 银行转账：不能出现金额不一致
- 库存扣减：不能超卖
- 特点：宁可不可用，也不能数据不一致

**AP场景：**
- 商品浏览：短时间数据不一致可接受
- 社交点赞：计数短暂差异不影响体验
- 特点：优先保证可用性，数据最终一致

## 二、BASE理论

### BASE的含义

| 维度 | 说明 |
|------|------|
| BA (Basically Available) | 基本可用：允许部分失败 |
| S (Soft State) | 软状态：数据允许中间状态 |
| E (Eventually Consistent) | 最终一致性：一段时间后数据一致 |

### ACID vs BASE

| 维度 | ACID | BASE |
|------|------|------|
| 一致性 | 强一致 | 最终一致 |
| 可用性 | 可能牺牲 | 高可用优先 |
| 适用场景 | 传统单机事务 | 分布式系统 |
| 实现代价 | 高 | 低 |

## 三、分布式事务方案

### 方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC | 强一致 | 差 | 中 | 传统数据库 |
| TCC | 强一致 | 中 | 高 | 资金类强一致 |
| Saga | 最终一致 | 好 | 高 | 长流程业务 |
| 本地消息表 | 最终一致 | 好 | 低 | 大多数业务 |
| 事务消息 | 最终一致 | 好 | 中 | 有MQ的场景 |

### 2PC (Two-Phase Commit)

**流程：**
```
准备阶段：
  -> 协调者问所有参与者"能否提交？"
  -> 参与者执行事务但不提交，返回Yes/No

提交阶段：
  -> 如果所有参与者都返回Yes，协调者发送提交
  -> 如果有参与者返回No，协调者发送回滚
```

**问题：**
- 性能差：同步阻塞，锁定资源
- 单点故障：协调者故障，参与者一直等待
- 数据不一致：协调者发送提交时部分参与者故障

### TCC (Try-Confirm-Cancel)

**三个阶段：**
```
Try：尝试执行，预留资源
  -> 订单服务：创建订单（状态=待确认）
  -> 库存服务：预扣库存（状态=冻结）
  -> 账户服务：预扣余额（状态=冻结）

Confirm：确认执行，真正提交
  -> 订单服务：订单状态=已确认
  -> 库存服务：扣减库存
  -> 账户服务：扣减余额

Cancel：取消执行，回滚资源
  -> 订单服务：订单状态=已取消
  -> 库存服务：释放冻结库存
  -> 账户服务：释放冻结余额
```

**特点：**
- 强一致性
- 性能比2PC好
- 业务侵入性强，需要实现三个接口

### Saga模式

**原理：**
- 长流程拆分为多个本地事务
- 每个本地事务有对应的补偿事务
- 前面事务失败，执行后面事务的补偿

**示例：**
```
正向操作：
  T1: 创建订单
  T2: 扣减库存
  T3: 扣减余额
  T4: 发送通知

补偿操作（如果T3失败）：
  C2: 恢复库存
  C1: 取消订单
```

**特点：**
- 最终一致性
- 适合长流程业务
- 需要设计补偿逻辑

### 本地消息表

**原理：**
```
Service A:
  1. 开启本地事务
  2. 执行业务操作
  3. 插入消息到本地消息表
  4. 提交事务

定时任务:
  1. 扫描本地消息表
  2. 调用Service B
  3. 成功则删除消息
  4. 失败则重试

Service B:
  1. 接收消息
  2. 执行业务
  3. 返回成功
```

**特点：**
- 最终一致性
- 实现简单，可靠性高
- 需要定时任务轮询

### 事务消息（RocketMQ）

**流程：**
```
1. 发送半消息：消息对消费者不可见
2. 执行本地事务
3. 提交/回滚消息：
   -> 事务成功：提交消息，消费者可见
   -> 事务失败：回滚消息，丢弃
4. 消息状态未知：反向查询事务状态
```

**特点：**
- 最终一致性
- MQ原生支持，可靠性高
- 依赖特定MQ（RocketMQ）

## 四、幂等性设计

### 为什么需要幂等

**场景：**
- 用户重复点击提交
- 网络重试导致重复请求
- 消息队列重复消费

**没有幂等的后果：**
- 重复下单
- 重复扣款
- 数据重复

### 幂等性实现方案

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| 唯一ID | 每个请求有唯一标识，处理前检查 | 通用 |
| 数据库唯一索引 | 插入数据时唯一约束冲突 | 新增数据 |
| 状态机 | 只有特定状态转换才允许 | 有状态的业务 |
| Token机制 | 请求前获取Token，使用后失效 | 表单提交 |

### 唯一ID实现

```java
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;

    @Autowired
    private IdempotentRepository idempotentRepository;

    public void createOrder(CreateOrderRequest request) {
        String requestId = request.getRequestId();

        // 检查是否已处理
        if (idempotentRepository.exists(requestId)) {
            return; // 已处理，直接返回
        }

        // 执行业务
        Order order = buildOrder(request);
        orderMapper.insert(order);

        // 记录已处理
        idempotentRepository.save(requestId);
    }
}
```

### Token机制

```java
@Service
public class TokenService {

    @Autowired
    private RedisTemplate redisTemplate;

    public String generateToken() {
        String token = UUID.randomUUID().toString();
        redisTemplate.opsForValue().set("token:" + token, "1", 5, TimeUnit.MINUTES);
        return token;
    }

    public boolean validateToken(String token) {
        String key = "token:" + token;
        Boolean deleted = redisTemplate.delete(key);
        return Boolean.TRUE.equals(deleted); // 删除成功才有效
    }
}
```

## 五、补偿机制与对账

### 补偿机制

**补偿场景：**
- 分布式事务最终一致性过程中出现异常
- 下游服务失败需要回滚
- 数据不一致需要修复

**补偿方式：**
- 自动补偿： Saga模式中的补偿事务
- 手动补偿： 运营后台手动修复
- 定时补偿： 定时任务扫描异常数据并修复

### 对账机制

**为什么需要对账：**
- 技术手段保证不了100%一致
- 网络故障、服务异常可能导致数据不一致
- 需要定期核对数据一致性

**对账流程：**
```
1. 从系统A导出数据
2. 从系统B导出数据
3. 比对差异
4. 生成差异报告
5. 手动或自动修复
```

**对账维度：**
- 日报： 每天核对前一天数据
- 实时对账： 关键业务实时监控
- 差异告警： 发现差异立即告警

## 六、Seata框架

### Seata模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| AT | 自动事务，无侵入 | 多数场景 |
| TCC | 手动实现三个接口 | 强一致要求 |
| SAGA | 状态机编排 | 长流程业务 |
| XA | 标准XA协议 | 传统数据库 |

### AT模式使用

```java
@GlobalTransactional
public void purchase(String userId, String commodityCode, int orderCount) {
    // 扣减库存
    storageService.deduct(commodityCode, orderCount);

    // 创建订单
    orderService.create(userId, commodityCode, orderCount);

    // 扣减账户
    accountService.debit(userId, orderCount * 5);
}
```

**原理：**
- 一阶段： 本地事务提交，自动记录回滚日志
- 二阶段： 全部成功则删除日志，失败则根据日志回滚

## 七、面试高频题

### 1. CAP定理是什么？
分布式系统不能同时满足一致性、可用性、分区容错性 -> 网络分区必然存在，实际选择CP或AP -> CP强一致但可用性差，AP高可用但弱一致性

### 2. 分布式事务有哪些方案？
2PC强一致但性能差，TCC业务侵入高 -> Saga适合长流程，本地消息表最实用 -> 事务消息需要特定MQ支持 -> 按业务场景选择合适方案

### 3. 怎么保证接口幂等？
每个请求有唯一ID，处理前检查是否已处理 -> 数据库唯一索引约束 -> Token机制，Token使用后失效 -> 状态机，只有特定状态转换才允许

### 4. 本地消息表怎么实现？
Service A本地事务中同时写入业务数据和消息 -> 定时任务扫描消息表调用Service B -> 成功删除消息，失败重试 -> Service B需要幂等
